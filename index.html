Capabilities

Everything you need to eliminate surprises

Built for infrastructure teams managing hundreds of servers across multiple clients and geographies.

Real-time risk scoring

Every server receives a live failure probability score updated every 60 seconds. Scores account for memory, disk I/O, network latency, and error log patterns simultaneously.

  • 4-tier risk classification (Critical / High / Medium / Low)
  • 6.4-hour average warning before failure
  • Per-DC and per-client risk aggregation

Missing data resilience

Handles up to 25% missing telemetry using MICE imputation — predictions stay accurate even when log pipelines degrade.

Auto-adapts to new infra

Online learning calibrates the model to each client's unique workload patterns within 72 hours of onboarding. No manual tuning required.

Multi-client isolation

Fully tenant-isolated models and dashboards. Each client's data never influences another client's predictions.

Intelligent alerting

Priority-ranked alerts deduplicated across data centers. Suppresses noise during planned maintenance windows automatically.

Sub-2s prediction latency

Full ensemble inference (XGBoost + LightGBM) completes in under 2 seconds per server at P50. Hard SLA of 5 seconds at P99.

Data science approach

How the model was actually built

This dashboard visualizes the output of a real, reproducible ML pipeline. The training code lives in /ml alongside this site.

Problem framing

Binary classification per server per time window: will this server experience a failure event in the next 24 hours? Labels are derived from historical incident tickets; features are aggregated from rolling windows (1h / 6h / 24h) over memory, disk I/O, network latency, error rate, and crash counts.

  • Target: failure_within_24h (binary)
  • Grain: one row per server per hour
  • Class imbalance handled via scale_pos_weight + focal-loss variant

Feature engineering

Rolling means, std deviations, and rate-of-change (slope) per metric, plus categorical encodings for client and data center. See ml/features.py.

Validation strategy

Time-based split (train on months 1–5, test on month 6) plus per-client leave-one-out validation to check generalization across environments — not a random shuffle split, which would leak temporal information.

Model choice

XGBoost and LightGBM ensembled by weighted averaging. Gradient-boosted trees were chosen over deep learning for interpretability (SHAP values), fast inference (<2s/server), and strong performance on tabular, heterogeneous telemetry.

Honesty about the demo

The live dashboard above runs on client-side synthetic data (js/data.js) so it works instantly on GitHub Pages with zero backend. The /ml folder contains the actual trainable model — run it locally to reproduce the metrics shown, or swap in real telemetry.
